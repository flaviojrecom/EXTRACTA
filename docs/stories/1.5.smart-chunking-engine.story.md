## Story

**As a** developer building the EXTRACTA pipeline,
**I want** a smart chunking engine that splits structured documents into token-bounded chunks with semantic overlap,
**so that** the output is optimized for RAG retrieval with consistent chunk sizes, preserved structures, and rich metadata.

## Acceptance Criteria

1. [x] Chunking por seção semântica primeiro, depois por limite de tokens
2. [x] Token counting via js-tiktoken (cl100k_base default)
3. [x] Chunk size configurável: 500-1000 tokens (default: 750)
4. [x] Overlap semântico: última(s) frase(s) completa(s) do chunk anterior (não corte fixo)
5. [x] Sentence boundary detection funcional
6. [x] Nunca quebra no meio de: listas, tabelas, code blocks, parágrafos
7. [x] Metadados por chunk: `{ chunk_id, chapter, section, token_count, overlap_tokens, position }`
8. [x] Interface `ITokenizer` implementada para permitir alternativas futuras
9. [x] Testes de consistência: todos os chunks dentro do range configurado
10. [x] Benchmark: processar documento de 100k tokens em < 5 segundos

## Dev Notes

### Architecture References

Todas as interfaces seguem o documento `docs/architecture/pipeline-architecture.md`.

**Key Interfaces:**

```typescript
// src/chunker/types.ts (definida em Story 1.1)
interface ITokenizer {
  encode(text: string): number[];
  countTokens(text: string): number;
  name: string;  // e.g., "cl100k_base"
}

interface ChunkOptions {
  minTokens: number;       // default: 500
  maxTokens: number;       // default: 1000
  targetTokens: number;    // default: 750
  overlapStrategy: 'semantic' | 'fixed';
  preserveStructures: boolean;  // lists, tables, code blocks
}

interface Chunk {
  id: string;              // e.g., "chunk-001"
  content: string;         // Markdown content
  metadata: ChunkMetadata;
}

interface ChunkMetadata {
  chunkId: string;
  chapterTitle?: string;
  sectionTitle?: string;
  tokenCount: number;
  overlapTokens: number;
  position: {
    index: number;         // 0-based chunk index
    total: number;         // total chunks in document
  };
}
```

**Pipeline Stage:**

```typescript
// Chunker implementa:
IPipelineStage<StructuredDocument, Chunk[]>
// Input: StructuredDocument, Output: Chunk[]
```

**Data Flow:**
```
StructuredDocument → [Chunker] → Chunk[] (with metadata)
```

**Key Design Decision:** Chunk overlap usa sentence boundary (semantic) por default, não corte fixo de tokens. Melhor qualidade de retrieval para RAG.

**Depends on:** Story 1.1 (interfaces, chunk types), Story 1.4 (StructuredDocument como input)
